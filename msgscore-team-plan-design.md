## 1. 設計思想

### 2軸マトリクス

MsgScoreのB2B拡張は「機能パッケージ × 席数パッケージ」の2軸で設計する。

- **横軸（機能）**: Team / Team Pro
- **縦軸（席数）**: S(5席) / M(10席) / L(30席)

**個人開発者が営業なしで売るための3条件**:

1. 決裁者がクレカで即購入できる（稟議不要な金額帯）
2. チーム管理がシンプル（IT部門の介入不要）
3. 「チームで使う理由」が個人Proと明確に違う

### 機能パッケージの定義

| パッケージ | コンセプト | 一言で言うと |
|-----------|----------|------------|
| Team | チームで使える | 共有・管理・ガバナンス基本 |
| Team Pro | チームで賢くなる | 組織学習・自動化・API連携 |

### アップグレードパス

```
Free（お試し）
  │  「もっと使いたい」
  ▼
Pro（個人で本格利用）
  │  「チームメンバーも使い始めた」
  ▼
Team S（5人で共有+管理）
  │  「組織のナレッジを蓄積したい」
  ▼
Team Pro S（5人で組織学習）
  │  「人数が増えた」
  ▼
Team Pro M / L
```

どの段階からでも直接申し込み可能。いきなりTeam Pro Mから始めるチームもあり得る。

## 4. Team固有機能の詳細設計

### 4.1 チームワークスペース

**チーム作成フロー（セルフサーブ）**:

1. Pro会員がTeamにアップグレード or 新規でTeam申し込み
2. チーム名を入力
3. メンバーをメールアドレスで招待
4. 招待メール「○○さんがMsgScoreのチームに招待しました」→ ワンクリック参加

**ロール設計（2種類のみ）**:

| ロール | できること |
|--------|----------|
| 管理者（Owner） | メンバー招待/削除、請求管理、ブランドボイス設定、共有プリセット管理、最低スコアルール設定 |
| メンバー（Member） | スコアリング、フィードバック投稿、共有プリセット利用、履歴閲覧 |

管理者は最初にチームを作った人がデフォルト。追加管理者は+500円/人/月のアドオン。

### 4.2 ブランドボイス設定（Team以上）

管理者のみ編集可能。

**設定項目**:

| 項目 | 内容 | 例 |
|------|------|-----|
| トーン指針 | 自由記述。組織の配信トーン | 「丁寧だが堅すぎない。です・ます調。親しみやすさと品格のバランス」 |
| NGワード | スコア減点対象の単語リスト | 最安値, 業界No.1, 絶対, 確実, 保証, 激安 |
| 必須チェック | 記載必須の要素 | 配信停止リンク、問い合わせ先、ブランド名 |
| 件名ルール | 件名の書式ルール | 【】墨付きカッコ不使用、絵文字1つまで |

**スコアリングへの反映（プロンプト注入）**:

```
## ブランドボイス設定（管理者が設定済み）

【トーン指針】
{brandVoice.tone}

【NGワード】以下の単語が含まれる場合、配信適正スコアを減点:
{brandVoice.ngWords}

【必須チェック】以下が欠けている場合、配信適正スコアを大幅に減点:
{brandVoice.requiredChecks}

【件名ルール】
{brandVoice.subjectRules}
```

### 4.3 最低スコアルール（Team以上）

管理者が設定。

| 設定項目 | 選択肢 |
|---------|--------|
| 最低スコアライン | 任意の数値（推奨65） |
| 未達時のアクション | 警告表示のみ / 警告+「要修正」バッジ / 警告+Slack通知（管理者宛）※Slack通知はTeam Proのみ |

スコアが最低ライン未満の場合、結果画面に警告を表示:

```
⚠️ チームの最低スコアライン未達
スコア 58/100 は、チームの最低ライン 65 を下回っています。
改善案を参考に修正してください。
```

### 4.4 チーム利用統計ダッシュボード（Team以上）

管理者が閲覧可能。CSVエクスポート対応。

**表示項目**:
- スコアリング実行数（月次）
- チーム平均スコア（先月比の増減）
- 最低スコアライン未達率（改善傾向の可視化）
- メンバー別: 実行数、平均スコア、最頻利用目的
- スコア推移グラフ（折れ線、月次）
- 個人 vs チーム平均比較

## 5. Team Pro 固有機能の詳細設計

### 5.1 上司フィードバック収集

**フロー**:

1. メンバーがスコアリング実行
2. 結果画面の「フィードバックを依頼」ボタンで共有リンク生成
3. 上司がリンクを開いてフィードバックを入力（アカウント不要）
4. フィードバックがDBに保存 → 5件以上で自動要約 → 組織ナレッジに蓄積

**フィードバック入力画面**:

| 項目 | 入力方式 |
|------|---------|
| 立場 | 選択: 部長 / 課長・マネージャー / リーダー・先輩 |
| AIスコアの評価 | 選択: 妥当 / やや高い / やや低い / 大幅にずれている |
| 修正スコア | 数値入力（上司の体感値） |
| 指摘事項 | 自由記述 |
| 対象軸 | 複数選択: 開封誘引力 / 読了性 / CTA強度 / ターゲット適合度 / 配信適正 |

**重要**: 上司はMsgScoreのアカウントがなくてもフィードバック可能。共有リンクを開くだけ。これがボトムアップ導入の鍵。

### 5.2 組織ナレッジ自動蓄積

**蓄積の仕組み**:

1. フィードバックが5件以上溜まる
2. Claude APIで自動要約（「これらのフィードバックから組織の評価傾向を抽出して」）
3. 役職タグ付きで `organization_knowledge` テーブルに保存
4. 次回スコアリング時にシステムプロンプトに注入

**プロンプト注入**:

```
## 組織固有の評価基準（実績フィードバックから学習済み）

【部長レベルの指摘傾向（8件のフィードバックから）】
- 品質・ストーリー訴求を価格訴求より高評価
- 【】墨付きカッコは自社ブランドにNG
- 件名20文字以内が社内ルール
- 既存顧客向けは「特別」より感謝型を好む

【課長レベルの指摘傾向（5件のフィードバックから）】
- LINE配信の絵文字は1つまで
- CTAは「詳しくはこちら」より具体的動詞
- 午前配信の方が開封率高い傾向あり

上記は組織の上長からのフィードバックに基づく実績ベースの基準です。
汎用的な評価基準と矛盾する場合は、組織基準を優先してください。
ただしマーケティングの原則に明らかに反する場合は、その旨を改善提案で指摘してください。
```

### 5.3 配信実績CSVインポート

**CSVフォーマット（テンプレート提供）**:

```csv
date,channel,subject,body,recipients,open_rate,ctr,cv_count,cv_type
2026-01-15,email,【限定】春コスメ50%OFF,,10000,23.4,4.7,47,purchase
2026-01-25,line,,🌸春コスメ特集！今だけ全品送料無料✨,8000,62.1,8.3,94,purchase
```

**必須カラム**: subject, channel
**任意カラム**: date, body, recipients, open_rate, ctr, cv_count, cv_type

**インポート後の処理**:

1. パース + バリデーション（Shift-JIS自動対応）
2. プレビュー画面（件数、チャネル内訳、期間、平均値表示）
3. Claude APIで傾向分析（バッチ処理）
4. 分析結果を `organization_knowledge` に保存

**分析結果の例**:

```
- 件名に「限定」を含む配信: 平均開封率26.3%（6件）
- 件名に「お知らせ」: 平均開封率18.1%（5件）
- 感謝型件名: 平均開封率27.5%（3件）
- 最高CTR件名: 「あなたへの特別なご案内」(6.8%)
- 配信曜日別: 火曜が最高（開封率+3.2pt）
```

**定期バッチ**: Team Proでは月次の自動再取込にも対応（CSVを所定の場所にアップロード）。

### 5.4 承認フロー

管理者が設定。最低スコアルール未達の配信に対して承認を要求。

**フロー**:

1. メンバーがスコアリング → 最低ライン未達
2. Slack/Teams通知が管理者に届く
3. 管理者が内容確認 → コメント付きで「修正依頼」or「承認」
4. 承認されると「配信OK」バッジが付く

### 5.5 フィードバック傾向の可視化

メンバー向け画面。上司の指摘パターンをカテゴリ別に表示。

```
松本部長の指摘傾向:
  ①丁寧さ・品格 (38%)
  ②CTA具体性 (28%)
  ③ブランド整合 (20%)
  ④その他 (14%)
```

### 5.6 指摘対応ステータス管理

フィードバックで受けた指摘を「未対応/対応済み」でトラッキング。

```
今月の指摘対応状況:
  ✅ NGワード使用: 対応済み（3件→0件に改善）
  ⚠️ CTA不明確: 1件未対応
  ✅ 件名文字数超過: 対応済み
```

## 6. 実画面プレビュー機能（Pro以上共通）

### 6.1 メール件名プレビュー

**モバイル（iPhone受信トレイ）**: 件名+プレビューテキストの表示。前後にダミーメール。件名が切れる場合に警告。

**デスクトップ（Gmail）**: 差出人+件名+プレビューテキストの1行表示。

**表示される情報**:
- 件名の文字数と切れるかどうかの判定
- モバイル推奨文字数（20文字以内で全文表示）
- プレビューテキストの表示範囲

### 6.2 LINE配信プレビュー

**トーク画面**: LINEのトークUIを模した表示。吹き出し内にテキスト。文字数カウント（500文字上限）。

**トーク一覧**: ブランド名+冒頭40文字の表示。実際の一覧での見え方を再現。

### 6.3 なぜ刺さるか

テスト配信を1回減らせるだけで、1配信あたり10-15分の時短。月20配信なら月4-5時間の削減。チーム稟議で使える数字。

## 8. DB設計（Supabase）

```sql
-- チーム
create table teams (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  created_by uuid references auth.users not null,
  plan text not null,          -- 'team_s' / 'team_m' / 'team_l' / 'team_pro_s' / 'team_pro_m' / 'team_pro_l'
  max_seats int not null,      -- 5 / 10 / 30
  stripe_subscription_id text,
  created_at timestamp default now()
);

-- チームメンバー
create table team_members (
  id uuid primary key default gen_random_uuid(),
  team_id uuid references teams not null,
  user_id uuid references auth.users not null,
  role text not null default 'member',  -- 'owner' / 'member'
  invited_email text,
  status text not null default 'pending',  -- 'pending' / 'active' / 'removed'
  joined_at timestamp,
  created_at timestamp default now(),
  unique(team_id, user_id)
);

-- 共有セグメントプリセット
create table team_presets (
  id uuid primary key default gen_random_uuid(),
  team_id uuid references teams not null,
  name text not null,
  segment jsonb not null,  -- AudienceSegment（conversionGoal含む）
  created_by uuid references auth.users not null,
  created_at timestamp default now()
);

-- スコア履歴（個人+チーム共用）
create table score_history (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references auth.users not null,
  team_id uuid references teams,  -- NULLなら個人
  channel text not null,
  input_text text not null,
  subject text,
  audience jsonb not null,
  result jsonb not null,           -- ScoreResponse全体
  actual_open_rate decimal,        -- 配信後の実績入力
  actual_ctr decimal,
  actual_cv_count int,
  created_at timestamp default now()
);

-- フィードバック
create table feedbacks (
  id uuid primary key default gen_random_uuid(),
  score_history_id uuid references score_history not null,
  team_id uuid references teams not null,
  reviewer_role text not null,     -- '部長' / '課長' / 'リーダー'
  score_agreement text not null,   -- 'agree' / 'slightly_high' / 'slightly_low' / 'far_off'
  adjusted_score int,
  comment text not null,
  target_axes text[],              -- ['開封誘引力', 'CTA強度']
  resolution_status text default 'open',  -- 'open' / 'resolved'
  created_at timestamp default now()
);

-- 組織ナレッジ（自動蓄積）
create table organization_knowledge (
  id uuid primary key default gen_random_uuid(),
  team_id uuid references teams not null,
```

## 13. 最終評価サマリー

### コンサルタントの総合判断

| プラン | 当たり前 | テンション | ロックイン | 業務変革 | 総合 |
|--------|---------|----------|----------|---------|------|
| Free | ✅ | ✅ | ⚠️→✅ | ⚠️ | 合格。Pro転換導線OK |
| Pro | ✅ | ✅ | ✅ | ✅ | 大幅改善。満足度高い |
| Team | ✅ | ✅ | ✅ | ✅ | 前回最大課題を解決 |
| Team Pro | ✅ | ✅✅ | ✅✅✅ | ✅✅ | 文句なし |

### 設計の要点

1. **Teamにガバナンス基本を入れた**: 管理者が「管理できる」ことがTeamの存在意義。NGワード・ブランドボイス・最低スコアルールはTeamの「当たり前品質」
2. **Proに個人ガバナンスを入れた**: NGワード(10語)・スコア推移・実績入力でPro単体の満足度を大幅向上
3. **共通警告ワード5語は全プラン共通**: 「最安値/業界No.1/絶対/確実/保証」はスコアリングエンジンの一部としてFreeでも検出。スコアの信頼性（景表法リスクを見逃さない）はツールの根幹であり、課金壁の外に置く。Proでカスタム10語追加、Teamで組織一括管理
4. **Team Proは「学習+自動化」に集中**: 使うほど賢くなる（フィードバック蓄積、CSV実績）+ 手が離せる（承認フロー、Slack通知、API）。時間軸で価値が増す
5. **Freeに履歴3回・URLシェアを追加**: Pro転換導線として機能。「もっと見たい」がPro課金の動機に
6. **管理者アドオン(+500円/人/月)**: Team/Team Pro共通。管理権限の拡張はシンプルな従量課金

### Team Pro の ROI

月16,800円（10席）の投資に対し:
- 部長のレビュー時間: 月10時間→3時間（7時間削減 = 時給5,000円で35,000円相当）
- メンバーの配信品質: 平均スコア+8pt向上（3ヶ月）
- コンプラ事故: ゼロ（承認フロー+NGワード）
- **ROI: 2倍以上**
