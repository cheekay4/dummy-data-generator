# Session 14: ペルソナUXテスト — MsgScore（Phase 1〜2.5）

## テスト対象
MsgScore（https://msg-scorer.vercel.app）
Phase 1-A〜2.5 の全機能。

## テストの進め方
1. 各ペルソナ（通常5人 + 意地悪4カテゴリ）で全操作をウォークスルー
2. 発見された課題を一覧化
3. PO佐藤がビジネス観点でスクリーニング + フラグ付け
4. UI/UXスペシャリスト村上が全画面を操作しレビュー + 全課題に判定
5. PO佐藤が最終承認
6. 🔴判定（PO承認済み）の修正プロンプトを作成

## UI/UXスペシャリスト：村上 彩（むらかみ あや）

元Yahoo! JAPAN UXリサーチャー → freee デザインマネージャー →
SaaS/業務ツール専門のUXコンサルタント独立。BtoB SaaSのオンボーディング改善を得意とし、
日本市場向けツールのローカライズUXに精通。50以上のSaaS製品のUXレビュー実績。

持論: 「BtoBツールこそ初回体験が全て。2回目はない。操作に迷った瞬間に信頼は崩れ、二度と戻ってこない」

専門領域: SaaS/BtoBツールのオンボーディングUX、課金導線設計、ダッシュボードUI

レビュー基準:
| 観点 | チェック内容 |
|------|------------|
| 情報設計 | ページを開いて3秒以内にツールの目的が伝わるか |
| 操作フロー | 入力→実行→結果のフローに無駄なステップがないか |
| フィードバック | ボタン押下後に何が起きたか即座に分かるか |
| エラー回復 | エラー時にユーザーが自力で次のアクションを取れるか |
| モバイル対応 | スマホでの主要操作（入力→実行→結果閲覧）が快適か |
| アクセシビリティ | コントラスト比、フォーカス順序 |
| ブランド一貫性 | ページ間でUI言語・トーン・レイアウトが統一されているか |
| 生成待ちUX | ローディング演出が適切で、キャンセル可能か |
| APIエラー | ネットワーク切断・APIダウン時の表示が適切か |
| 信頼性表示 | AI生成結果の免責表示が適切か |
| 課金導線 | 無料枠使い切り後のUXが押し付けがましくないか |
| Pro機能ゲーティング | Freeユーザーへのアップグレード誘導が自然か |
| チーム機能 | 管理者/メンバーの権限分離が明確か |

判定基準:
| 判定 | 基準 | アクション |
|------|------|-----------|
| 🔴 修正する | ユーザーが目的を達成できない、または重大な混乱を招く | 今すぐ修正 |
| 🟡 後日対応 | 改善の余地はあるが、現状でも使える | バックログに記録 |
| ⚪ 修正しない | 好みの範囲、または工数に対して効果が小さい | 記録のみ |

## プロダクトオーナー：佐藤 健一（さとう けんいち）

元リクルート プロダクトマネージャー → SaaS スタートアップCPO →
個人開発プロダクトのアドバイザリー歴5年。
100以上のマイクロSaaSの立ち上げ・グロースに関与。

持論: 「完璧を目指して遅れるより、80点で出して市場の反応を見ろ。
ただし、信頼を失う20%だけは絶対に落とすな」

判断基準:
- 収益直結の課題か（課金フロー障害、離脱ポイント）→ 最優先
- ユーザーの信頼に関わるか（スコア表示の不整合、データ消失）→ 最優先
- 初回体験を損ねるか（初見で迷う、目的達成できない）→ 高優先
- 体験を磨く改善か（あると嬉しい、デザイン微調整）→ 後日
- 特定条件でしか発生しないか（レアケース）→ 後日 or 対応しない

## 意地悪ペルソナ

**⑥ 不正入力テスター（カテゴリA）**
- 空文字でスコアリング実行 → バリデーションエラー表示
- テキスト欄に `<script>alert(1)</script>` を入力 → エスケープされる
- テキスト欄に10,000文字の長文を入力 → 文字数制限 or 適切に処理
- テキスト欄に絵文字のみ「🎉🎉🎉」 → スコアリングが適切に動作
- テキスト欄に英語のみ → 日本語ツールとして適切な応答
- 配信母数に「-100」「0」「abc」「9999999999」を入力 → バリデーション
- セグメントスライダーの合計を100%以外にする → バリデーション
- NGワード設定に空文字/重複/特殊文字 → バリデーション
- カスタムセグメント名に空文字/100文字超 → バリデーション

**⑦ 課金フローテスター（カテゴリD）**
- Freeで5回使い切り → アップグレード導線表示の確認
- 6回目のスコアリング試行 → 429エラー + 適切なメッセージ
- Stripe Checkout画面でブラウザバック → アプリに戻れるか
- Pro課金成功後 → プランがすぐproに反映されるか
- Pro機能（プレビュー、CSVエクスポート等）がFreeで適切にゲーティングされているか
- 各ゲーティングUI（ぼかし+オーバーレイ / 非活性+ツールチップ）の表示
- Stripe Customer Portalでプラン管理ページが開けるか
- Team課金フロー → チームセットアップ画面への遷移
- 料金ページの表記: Free「レポート出力（PDF/CSV）」vs Pro「PDF/CSVレポート出力」の表記揺れ ← 既知

**⑧ AI出力品質テスター（カテゴリE）**
- 意味不明な入力（「あああああ」「asdfghjkl」）→ AIの応答品質
- 英語のみの入力 → 日本語でスコアリング結果が返るか
- 非常に短い入力（「セール」の2文字のみ）→ 適切にスコアリングされるか
- スコアリング中にページをリロード → 状態の復帰
- スコアリング中にネットワーク切断 → エラー表示とリトライ導線
- ローディング演出（5段階ステップ）の自然さ
- AIのJSON出力がパース失敗した場合のフォールバック表示
- APIタイムアウト（10秒超）時の「もう少しお待ちください」表示
- A/B案のテキストが入力のチャネルと整合しているか（件名なのに長文、等）

**⑨ チーム機能テスター（MsgScore固有）**
- Team未所属のユーザーが /team にアクセス → 適切なリダイレクト or メッセージ
- メンバーがブランドボイス設定を編集しようとする → 権限エラー
- メンバーがメンバー管理ページにアクセス → 権限エラー
- 席数上限まで招待済みの状態で追加招待 → エラー表示
- 招待リンクの有効期限（期限切れリンクの挙動）
- 削除されたメンバーがチームページにアクセス → 適切にブロック
- ブランドボイス設定のNGワードがスコアリング結果に実際に反映されるか
- 最低スコアルール未達時の警告バナー表示
- 修正依頼の送信・受信・対応済みフローの一連の動作
- チームプリセットの保存（20個上限）と呼び出し
- メンバーのスコア履歴が他チームメンバーから見えるか（見えるべき）
- 別チームのデータにアクセスできないか（RLS検証）

## テスト実施手順

### Step 1: ペルソナウォークスルー
全ペルソナ（①〜⑨）で実際に操作し、以下を記録:
- ✅ 目的達成できた
- ⚠️ 達成できたが迷った（どこで迷ったか）
- ❌ 達成できなかった（何が障害か）

### Step 2: 課題一覧
| # | 画面/機能 | ペルソナ | 課題 | 深刻度 |
|---|----------|---------|------|--------|

### Step 3: POスクリーニング
PO佐藤がビジネス観点で全課題をスクリーニング。
ビジネスクリティカルな課題にフラグ付け。明らかに対応不要な課題をフィルタ。

### Step 4: スペシャリストレビュー
スペシャリスト村上が自ら全画面を操作 + Step 2の課題に判定を付与。
POフラグ付き課題は重点レビュー。独自発見の課題も追加。

追加レビュー観点（MsgScore固有）:
- LP → スコアリング → 結果 → 履歴 の導線の自然さ
- Free → Pro のアップグレード体験（押しつけがましくないか）
- チームダッシュボードの管理者/メンバー切替の明快さ
- ブランドボイス設定 → スコアリング反映の体験（設定が効いてる感）
- 料金ページのFree/Pro/Team S/M/L の情報量と比較しやすさ

### Step 5: PO最終承認
| # | 課題 | 深刻度 | スペシャリスト判定 | PO承認 | PO理由 |
|---|------|--------|-------------------|--------|--------|

### Step 6: 🔴判定（PO承認済み）の修正プロンプト作成
